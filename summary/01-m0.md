# summary/01 — M0 完了概要

Issue #1 / PR #2（マージ済み）。ローカルで動く最小オブジェクトストアの心臓部。

## 何を作ったか

`src/lib.rs` の `ObjectStore`。Bitcask 型「追記ログ(WAL) + インメモリ index」設計:

| API | 動作 |
|-----|------|
| `open(dir)` | ログを頭から replay して index と write_offset を復元 |
| `put(bucket, key, value)` | レコードをログ末尾に append、value 位置を index に記録 |
| `get(bucket, key)` | index で位置を引き、ログを seek して読む |
| `delete(bucket, key)` | tombstone レコードを append、index から削除 |

ログのレコード形式: `[flags:u8][key_len:u32 LE][value_len:u32 LE][key][value]`（ヘッダ固定長9バイト）。

## 設計の勘所

- **全書き込みがシーケンシャル append** — ランダム書き込みを避けて速くする（ハードの気持ちに沿う）。
- **index はメモリ上の HashMap** — GET は「index で位置を引く→1回 seek→読む」だけ。
- **上書き/削除は追記で表現** — 古いデータはログに残る。回収は M5 の compaction のネタ。
- **replay は破損末尾を安全に打ち切る** — 各レコードが実ファイル長 `end` に収まるか検証。
  巨大 len ヘッダによる OOM と、範囲外レコードの index 登録を防ぐ。

## 品質

- `cargo test` 5件緑: put/get・上書き・delete後get・再起動復元・破損末尾の無視。
- `cargo clippy` 警告ゼロ。
- reviewer サブエージェントの 🔴 must（replay の入力未検証）と 🟡（key 衝突・UTF-8 厳密化）を解消。

## 意図的な非スコープ（後続マイルストーン）

- HTTP / S3互換 API
- **fsync による耐久性保証**（現状クラッシュで直近書き込みを失う可能性 → M1 以降で検討）
- 計測ベンチ（M1）
- ネットワーク / 分散（M2〜）
- log compaction / zero-copy（M5）
