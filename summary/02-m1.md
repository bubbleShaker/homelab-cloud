# summary/02 — M1 完了概要

Issue #6 / PR #7（マージ済み）。M0 の `ObjectStore` を「数字で殴れる」状態にする計測基盤。

## 何を作ったか

依存クレートゼロの自作ベンチ。計測ロジック（純粋）とワークロード実行（I/O）を2ファイルに分けた。

| ファイル | 責務 |
|---------|------|
| `src/metrics.rs` | レイテンシ列 `[Duration]` から要約統計を出す純粋関数群。I/O を持たない。 |
| `src/bin/bench.rs` | ストアに対し `warmup → PUT → GET` を回し、op ごとの所要時間を計測して表を出力。 |

### `src/metrics.rs`

- `LatencySummary::from_samples(&mut [Duration])` — count/min/max/mean と p50/p90/p99/p999 を返す。
  破壊的ソートで余計なコピーを避ける。空なら `None`。
- 分位点は **nearest-rank 法**: `rank = ceil(p/100 * N)` を `1..=N` にクランプして 0-indexed で引く。補間はしない（挙動がブレず説明しやすい定義を選んだ）。
- 平均は `u128` ナノ秒の総和経由で算出（`Duration / u32` の件数オーバーフローを回避）。
- `throughput(count, elapsed)` — ops/sec。経過時間 0 は 0 を返す（0除算回避）。

### `src/bin/bench.rs`

- `warmup（使い捨ての別ストア）→ PUT フェーズ → GET フェーズ` の順で計測。
- op ごとに `Instant::now()`〜`elapsed()` を採り、フェーズ全体の経過は別途 throughput に使う二本立て。
- CLI: `--ops`（既定 100000）/ `--value-size`（既定 256B）/ `--dir`（既定は OS 一時ディレクトリ）。
- 実行例: `cargo run --release --bin bench -- --ops 50000 --value-size 256`

## 設計の勘所

- **計測ロジックと I/O を分離** — `metrics` は `bench` に依存しない内向き依存。percentile/throughput の正しさを `cargo test` で固定でき、SRP も満たす。
- **依存クレートを増やさない（criterion 不採用）** — 計測を自分で持つことで、M5 の削り込みで「どこがどれだけ速くなったか」を自分のコードで追える。M0 の依存ゼロ方針も継承。
- **既定の data_dir は Linux ネイティブ FS（`std::env::temp_dir()`＝WSL の ext4）** — `/mnt/c`（Windows 側）を指定すると drvfs/9p のオーバーヘッドを測ってしまい、ストア本体の数字にならない。`--dir` で上書き可。
- **fsync なしの数字であることを出力に明記** — M0 は fsync 非スコープ。PUT のスループットは「OS のライトバックキャッシュに載せるまで」の値で、電源断でディスクに残る保証はない。読み違え防止のため注記を出す。
- **warmup は使い捨ての別ストアで実行** — アロケータ/分岐予測/CPU 周波数を温めつつ、計測対象ストアの index は空のまま保つ（warmup のレコードを混ぜない）。
- **`black_box`** — GET の戻り値 `Option<Vec<u8>>` を `std::hint::black_box` に渡し、最適化で GET 自体が消えて計測が無意味になるのを防ぐ。PUT はログ追記の副作用があるので不要。

## 計測結果の例と読み方

WSL ext4 / value 256B / 5万件 / release ビルド:

| フェーズ | throughput | p50 | p99 | p99.9 | max |
|---------|-----------|-----|-----|-------|-----|
| PUT | 約 51万 req/s (125 MiB/s) | 1.7µs | 4.3µs | 16.9µs | 2.3ms |
| GET | 約 62万 req/s (151 MiB/s) | 1.5µs | 2.1µs | 14.6µs | 658µs |

- **GET が PUT より速い**のは、GET が index ヒット→1 seek の読みだけなのに対し、PUT はログ末尾への append（+ 時々ファイル拡張）を伴うため。
- **PUT の max 2.3ms** はファイル拡張やページ確保が走った瞬間のスパイク。p99 と max の乖離が「テールの正体」で、M5 で削るネタになる。
- これらは fsync なしの数字。耐久性を入れると PUT は桁で落ちるはずで、その比較が後続の題材。

## 意図的な非スコープ（後続マイルストーン）

- **fsync 有無の比較**（耐久性 ON/OFF でスループットがどう変わるか）
- **並行クライアント**（今は単一スレッドで基礎数字。並行負荷は M2 以降）
- **グラフ描画**（今は stdout の表まで）
- ネットワーク / 分散（M2〜）、log compaction / zero-copy（M5）

物理の入口に立つ。M3 で M0 の自作S3を実ハードに載せ、この M1 のベンチを LAN 越しに回して数字を取り直す。
