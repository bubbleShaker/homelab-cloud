# knowledge/01 — M0 コードの「分かりにくいかも」Q&A

M0（`src/lib.rs`）で、読み手が引っかかりやすい箇所を質問形式で解説する。
「全部理解しないといけない」わけではないのだ。std に閉じたこの層は信頼してよい境界。
気になった項目だけ拾えばいいのだ。

図で見たい人は [`02-m0-diagrams.md`](./02-m0-diagrams.md)（UML / mermaid: クラス図・シーケンス図・replay フロー）へ。

## Q1. なぜヘッダを固定長9バイト＋little-endian にしたのか？

`[flags:1][key_len:4][value_len:4]` の9バイト固定。

- **固定長にする理由**: レコードの位置計算が足し算だけで済む。
  `value_offset = レコード開始 + 9 + key_len`、`次レコード = value_offset + value_len`。
  可変長ヘッダだと「ヘッダが何バイトか」を別途読む必要が出て複雑になる。
- **little-endian にする理由**: 数値をバイト列にする順序を固定して、
  別マシン（Pi など)でログを読んでも同じに解釈されるようにするため。
  `to_le_bytes()` で書き、`from_le_bytes()` で読む。ペアで使うのが鉄則。

これは m5stack でフレームバッファのバイト配置を意識したのと同じ「表現を固定して速く単純にする」発想。

## Q2. なぜ O_APPEND(追記モード)を使わず、offset を自前管理しているのか？

`open()` は追記モードを使わず、read+write で開いて `write_offset` を自分で持っている。

- 追記モード(O_APPEND)は「書くと必ず末尾に行く」が、**GET では途中を読みたい**（seek する）。
  1本のハンドルで「途中を読む」と「末尾に書く」を混ぜると、追記モードは相性が悪い。
- だから「末尾位置 = `write_offset`」を自分で覚えておき、
  PUT は `seek(write_offset)`→write、GET は `seek(値の位置)`→read、と明示的に seek する。
- 単一スレッド前提だから成立する素朴な設計。並行対応は将来の課題。

## Q3. `let Some(loc) = self.index.get(...).copied() else { return Ok(None); }` は何？

Rust の **let-else** 構文。「パターンにマッチしたら束縛、しなければ else を実行」。
ここは「index に key があれば `loc` に取り出す、無ければ即 `Ok(None)` で返す」。
`if let ... { } else { }` のネストを避けて、早期リターンを平らに書ける。

`.copied()` は `Option<&ValueLoc>` を `Option<ValueLoc>` にする（参照を値のコピーに変える）。
`ValueLoc` は `Copy`（8+4バイトの小さな値）なので安全に丸ごとコピーできる。

## Q4. なぜ GET が `&mut self`（可変借用）なのか？ 読むだけなのに。

`self.log.seek(...)` がファイルの**読み書きカーソルを動かす**＝ File の内部状態を変える操作だから。
Rust では状態を変えるメソッドは `&mut self` を要求する。「論理的には読み取り」でも、
seek でカーソルという状態を書き換えているので可変借用が要る、というのがポイント。

## Q5. 削除がなぜ「その場で消す」ではなく tombstone の追記なのか？

追記ログは**末尾に足すことしかしない**設計（シーケンシャル書き込みで速いから）。
途中のレコードを物理的に消すとランダム書き込みになり、設計思想に反する。
だから削除は「消したよマーカー(tombstone)」を末尾に足し、index からキーを外すだけ。
古い実体はログに残り続ける。それを回収するのが将来の **compaction**（M5）。

## Q6. replay の `record_end > end` で break、が「安全な打ち切り」になる理屈は？

`end` は実ファイルの長さ。`record_end = offset + 9 + key_len + value_len` は
「このレコードが本当なら、ここまでファイルがあるはず」という終端。

- もし `record_end > end` なら、**ファイルの実長を超える**＝そのレコードは途中で切れている
  （クラッシュ時の部分書き込み）か、壊れた len。→ そこで replay を打ち切る。
- これで `vec![0u8; key_len]` に巨大値が来て **OOM する事故を防ぐ**（key_len は必ず end 以下に収まる範囲でしか読まない）。
- 打ち切った地点が `write_offset` になるので、**次の PUT がゴミを上書き**して自然に修復される。

reviewer に 🔴 で指摘されたのがまさにこの検証の欠如。「入力を信用しない」の基本。

## Q7. `compose_key` の区切りが `/` ではなくヌル文字(`\0`)なのはなぜ？

`/` 区切りだと `("a", "b/c")` と `("a/b", "c")` がどちらも `"a/b/c"` になり**衝突**する。
ヌル文字は通常のキー名に現れないので、区切りに使えば境界が曖昧にならない。
（S3 でも bucket 名に `/` は使えない等の制約があり、境界を守るのは重要。）

## Q8. テストの `TempDir` が Drop で後始末しているのは何のパターン？

**RAII**（Resource Acquisition Is Initialization）。
「値が生きている間だけリソースを持ち、スコープを抜けて破棄されるとき後始末する」。
`impl Drop for TempDir` に `remove_dir_all` を書くと、テスト終了時（パニックしても）に
一時ディレクトリが自動で消える。手動 cleanup の書き忘れを防ぐ Rust の定石。

## まとめ：信頼してよい境界

- この M0 は **std のファイル I/O だけ**で完結している。外部 crate 無し。
- 中核の不変条件は「**index は常にログ上の最新値の位置を指す**」の一点。
  put/delete/replay がこの不変条件を保つように書かれている、とだけ押さえれば十分。
